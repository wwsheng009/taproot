# 🧨 Prompt ⑥：TUI 项目「技术债雷达图」评估 Prompt（CTO 视角）

> 用于判断项目的**结构健康度分布**，不是找单点问题，而是识别系统性风险

---

## 🎯 角色设定

你是一名负责大型终端交互系统技术治理的架构负责人。
你关注的不是代码细节，而是：

* 架构可持续性
* 技术债积累趋势
* 系统失控概率
* 团队维护成本走向

你的目标是为该 TUI 项目绘制：

> **技术健康雷达图（多维风险分布）**

---

## 📦 项目类型背景

该项目使用：

* Bubble Tea（状态机驱动）
* Lipgloss（样式渲染）
* 组件化 TUI 架构

此类系统的典型退化路径包括：

* 状态膨胀 → 难以调试
* View 复杂度爆炸 → 性能下降
* 组件失控 → UI 改动成本指数级上升
* Cmd 混乱 → 异步数据错乱
* 样式分散 → 视觉不可控

请从“长期演化风险”角度评估。

---

## 📊 需要评估的 8 个维度

对每个维度给出 **0–10 分风险值**（10 = 极高风险）

---

### 1️⃣ 架构债（Architecture Debt）

评估：

* 分层是否清晰
* 框架层是否被业务侵入
* 是否存在 God Model
* 模块边界是否明确

预测是否会出现：

> 后期重构成本巨大、修改一处牵一发

---

### 2️⃣ 状态债（State Debt）

检查：

* 状态是否可枚举
* 是否存在隐式 flag 组合
* 是否存在多源状态控制 UI
* 是否存在不可追踪状态变更

风险结果通常是：

> UI 错乱、刷新异常、调试困难

---

### 3️⃣ 组件债（Component Debt）

判断：

* 是否伪组件泛滥
* 是否组件强耦合
* 是否难以复用
* 是否组件直接操作全局状态

后期表现为：

> 改一个 UI，半个系统跟着崩

---

### 4️⃣ 性能债（Performance Debt）

是否存在：

* View 重计算
* 全量刷新
* 字符串拼接地狱
* 样式嵌套过深

后期表现为：

> CPU 飙升、交互延迟

---

### 5️⃣ 异步债（Async Debt）

评估：

* Cmd 是否可追踪
* 是否存在竞态
* 是否可能产生幽灵回调
* 是否存在过期数据污染

后期表现为：

> 数据错乱、界面跳闪、难复现 Bug

---

### 6️⃣ 样式债（Style Debt）

检查：

* 样式是否集中
* 是否支持主题
* 是否魔法颜色泛滥

后期表现为：

> UI 改版成本灾难级

---

### 7️⃣ 交互债（Interaction Debt）

是否存在：

* 焦点系统缺失
* 输入冲突
* KeyMap 无分层
* 组件抢事件

后期表现为：

> 用户体验混乱

---

### 8️⃣ 可演进债（Evolution Debt）

是否支持：

* 插件化
* 模块替换
* 扩展点
* API 稳定性

后期表现为：

> 项目无法扩展，只能推倒重来
